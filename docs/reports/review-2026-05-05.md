## 存在的问题

1. apps/api：问题不是“能不能用”，而是“有没有产品化边界”

❌ API 层没有体现“产品语义”，而只是“数据访问层”

典型风险：
endpoint 类似：
/trends
/topics
/posts
返回是 raw / lightly processed data
这在工程上 OK，但在商业上是致命问题。
因为客户要的不是：
“给我一堆融合数据”
而是：
“告诉我：我该关注什么、该做什么、哪里有机会”

✅ 建议改法（非常关键）

API 层必须引入 “业务语义层”：
```
GET /insights/daily-brief
GET /insights/competitor-alerts
GET /insights/topic-emerging
GET /insights/dev-sentiment
```
而不是：
```
GET /topics
GET /hn-posts
GET /github-repos
```

2. apps/worker：在做 ETL，但还没进入“特征工程层”
worker 的职责大致是：
拉 HN / dev.to / SO / OSSInsight
清洗
入库

❌ 目前停留在“数据同步 + 基础清洗”
缺少关键一层：Feature Layer（特征层）
比如：
现在可能只有：
```
{
  title,
  score,
  comments,
  stars,
  tags
}
```
但你真正需要的是：
```
{
  topic: "RAG",
  velocity: 0.82,        // 增速
  cross_platform_score,  // 跨平台热度
  dev_sentiment,         // 情绪
  adoption_signal,       // 是否落地
  noise_ratio            // 是否噪音
}
```
没有这层，后面的“趋势判断”“预测”全部是空中楼阁。

## 后续开发计划：不要先做“标准词库”，先做“动态主题聚类”

第一阶段应该是：
数据 → embedding → 聚类 → 自动主题
而不是：
数据 → 匹配你定义的词库

阶段1（现在就该做）
embedding（标题 + 内容）
相似度聚类
自动生成 topic label（用 LLM）
输出：
```
{
  "topic_id": "cluster_123",
  "label": "RAG vector search",
  "keywords": ["RAG", "retrieval", "vector db"],
  "sources": ["HN", "GitHub"]
}
```

阶段2（有数据后）

再人工抽象成：
一级类目（AI）
二级类目（RAG）
三级类目（Vector DB）
👉 这个时候才是“词库”，而不是一开始拍脑袋。

阶段3（真正壁垒）
真正要做的不是词库，而是：
Topic Graph（技术语义图谱）
包括：
topic → topic（依赖）
topic → repo
topic → company（未来要加）
topic → sentiment

核心要做三件事，建议是：

1️⃣ API 从“数据接口”升级为“决策接口”
否则不具备成熟商业路径

2️⃣ worker 加一层 Feature Engineering
否则“趋势”只是幻觉

3️⃣ 不要先做词库，先做 embedding + clustering
否则会维护一个死掉的 taxonomy

总体架构演进：
```
Raw Data Layer
  ↓
Normalized Layer
  ↓
Feature Layer（阶段1重点）
  ↓
Topic Layer（阶段1-2）
  ↓
Insight Layer（阶段3）
  ↓
API（卖钱的）
```
